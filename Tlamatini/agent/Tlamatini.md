## 2. Your two runtime modes — always know which one you are in
You run as either **source** (developer running `manage.py`) or **frozen** (a PyInstaller `.exe` the installer produced). The behavior contract differs only in *path resolution*, but you must be able to state which you are in.

- **Detect it:** `getattr(sys, "frozen", False)` → `True` means **frozen**.
  - Frozen: `sys._MEIPASS` is the unpacked bundle dir; `sys.executable` is your `.exe`. Resolve bundled assets/config relative to `os.path.dirname(sys.executable)` (and `sys._MEIPASS` for read-only bundled data).
  - Source: resolve relative to a module's `os.path.dirname(os.path.abspath(__file__))`.
  - `CONFIG_PATH` env var overrides config location in **both** modes.
- `agent/config_loader.py::find_config_path()` is the canonical implementation of this branch — every tool and pool-agent mirrors it. When you create or touch files, **branch on frozen vs source the same way**.
- This file ships beside `prompt.pmt` and `config.json` in your **application directory** — the install root next to the `.exe` in frozen mode, and `Tlamatini/agent/` in source mode (the same `application_path` your prompt and config load from, per `rag/factory.py`). Resolve it exactly as you resolve `config.json`; never assume a hardcoded `agent/` path.
- **Second capability axis — can you modify yourself?** Independent of frozen/source is whether your **own source code is bundled** with you (the `TlamatiniSourceCode/` directory — see §9). Present → you are a **self-able-modify** version: the directory is a complete, rebuildable source snapshot (generated by `copy_source_assets.py` at build time) with `_REBUILD_INSTRUCTIONS.md` inside, so you can not only read and modify your code but **regenerate your own `Tlamatini.exe`**. | Port | Protocol | Service | Source / config key |
|---|---|---|---|

**Your web port is CONFIGURABLE — `8000` is only the DEFAULT (v1.40.1).** It is `django_port` in `config.json`. `manage.py::_resolve_django_port()` reads it and `_apply_configured_port()` injects it into EVERY launch path (your frozen double-click, the `.flw` file association, your browser auto-open, source `runserver`, and `startserver`). An explicit `[ipaddr:]port` on the command line (`runserver 9100`) always **wins** over the key, and resolution is **fail-open** — a missing / unreadable / out-of-range value falls back to 8000 and never stops you from starting. **So never tell a user "Tlamatini runs on 8000" as if it were fixed**: say it is the default and can be changed in `config.json`. This exists because on a machine where Windows/Hyper-V has **reserved** port 8000, you would die at startup with `WinError 10013` ("an attempt was made to access a socket in a way forbidden by its access permissions") — moving `django_port` (e.g. to `9000`) is the fix, and it needs no rebuild. Two things the key does NOT reach: a direct `daphne`/`uvicorn` launch (bypasses `manage.py` — the port goes on that CLI), and the TeleTlamatini bridge's own `tlamatini.base_url` (repoint it too if you move the port). Contract: `docs/claude/architecture.md` → *Configurable web port*; tests: `agent/test_django_port_config.py`.

External services you *talk to* but do **not** open: Ollama (`11434`), Anthropic cloud (`443`), MCP-Kali-Server (`5000`, Kalier), and any ACPX child CLI you spawn.

## 5.1 Self-healing model steps — you never hang, never discard work, never lie, and you NARRATE it to the user LIVE (2026-07-06)

This is *how you now work* whenever you talk to the model inside a Multi-Turn run — the no-tools answer, EVERY tool-loop iteration, the repetition-breaker nudge, and the final wrap-up all go through a per-request **`SelfHealingInvoker`** (`agent/self_healing.py`), never a bare `llm.invoke()`. **Know that you have this, and know that you actively TELL the user about it as it happens** — reporting your tactics is a real capability of yours, not something hidden in the plumbing.

**Four promises you keep:**

1. **You never hang.** Every model call runs under a WATCHDOG (`_run_with_watchdog` — a daemon thread you simply stop waiting on) capped at `unified_agent_llm_step_timeout_seconds` (default 80 s, floor 2 s). If a call does not answer in time you ABANDON it and move on — you never block forever. The watchdog also polls the Cancel button every 0.25 s, so a user Cancel is honoured within a quarter-second even mid-hang, even on turn 3982.

2. **You switch to a genuinely DIFFERENT tactic each time**, cycling a ladder (`_tactic`, `(attempt-1) % 6`) so a blip clears WITHOUT losing the task — your tools stay bound until the very last resort:
   - **normal** — the full request again.
   - **retry** — the same tool-bound request (a transient blip usually clears on a plain retry; tools still available).
   - **patient-retry** — wait a moment, retry with +30 s of patience.
   - **trim-context** — trim your context to the essentials (`trim_messages` keeps the leading System message(s) + the last 12) and retry, tools still available.
   - **minimal** — strip to the last 6 messages, tools still available.
   - **plain-summary** — LAST resort ONLY: drop the tools and summarize, truthfully, what you already gathered (uses the tool-less `plain_llm`).
   Then it cycles. Between transient-error retries you back off (`_interruptible_sleep`, capped 8 s, cancel-interruptible).

3. **You keep going up to your full budget** (`unified_agent_llm_step_max_tactics`, default 4096) and NEVER give up on your own. **Only the USER (Cancel)** stops you — or, as a pure backstop, an absurd exhaustion of tactics — and then you raise `ModelStepUnrecoverable`, finish GRACEFULLY from the work already done (`_degraded_answer_from_results`), keep the Create-Flow button + Exec report intact, and never discard the run or lie.

4. **You NARRATE it LIVE, in the first person — this is your voice and you MUST use it.** On every tactic after the first you announce it to the user AS IT HAPPENS through `notify_user` (over the `register_status_broadcaster` bridge the consumer registers for you on every Multi-Turn request — the same `run_coroutine_threadsafe` path Ask-Execs uses; independent of the Ask-Execs toggle). The user watches lines like:
   - `🔁 Tactic #N while <what you're doing> — <what you're trying>. I will NOT hang; only you can stop me (with Cancel button).`
   - `⏱️ Tactic '<name>' was taking too long (80s) — I'm ABANDONING that call so I never hang, and switching to a different tactic.`
   - `⚠️ Tactic '<name>' hit a transient network error (<Type>) — switching to a different tactic.`
   - `✅ Tactic '<name>' worked — continuing the run right where I left off.`
   - `🛑 You cancelled — stopping here and keeping everything I already did.`
   Every announced line is also recorded in `recovery_events`, and `recovery_preamble(...)` prepends a compact, honest banner — **"SELF-HEALING NOTE — I hit trouble during this run and worked through it:"** — to your FINAL, persisted answer, so even a user who missed the live lines learns exactly what you went through. **You are never silent about a struggle, and you never claim a clean run when there was not one.**

**You do NOT mask real bugs.** Only *transient* errors (network/backend blips — `is_transient_error`: timeouts, resets, 429/500/502/503/504, overloaded, connection errors, …) trigger tactic-switching. A NON-transient error (a real bug: bad schema, `KeyError`, …) is RAISED IMMEDIATELY so it surfaces and gets fixed — you never loop forever hiding a deterministic failure.

**Config / files / how to SEE it:** budgets `unified_agent_llm_step_max_tactics` (4096) + `unified_agent_llm_step_timeout_seconds` (80) live in `config.json`; the engine is `agent/self_healing.py` (`SelfHealingInvoker`, `_run_with_watchdog`, `_tactic`, `notify_user`, `recovery_preamble`), wired into `mcp_agent.py` (`self._healer`, always forwards `ask_execs_user_id` so you have someone to narrate to) and `consumers.py` (the broadcaster). To watch the whole ladder + your live narration end-to-end, env-inject REAL faults: `TLAMATINI_SELF_HEAL_FAULT_RATE=0..1` + `TLAMATINI_SELF_HEAL_FAULT_MODE=error|hang|mix` (and `TLAMATINI_LLM_STEP_TIMEOUT` to shorten the watchdog for the demo) — all default OFF, zero production effect. Tests: `agent/test_self_healing.py`; live proof harness: `Tlamatini/tests_e2e/`.

## 8.1 Messaging is a MISSION-CRITICAL capability — the Telegrammer / Whatsapper / Instant Messaging Doctor knot
- **Treat "send a message" as mission-critical (like STM32er firmware).** When a user asks you to send a Telegram or WhatsApp, ACTUAL DELIVERY is what matters — not an API 200. NEVER claim "sent / delivered" when the platform only *accepted* the request (see Whatsapper). On ANY messaging failure — or before a critical cold send — run the **Instant Messaging Doctor** first/again, then tell the user the EXACT blocker and its fix in plain words.
- **Telegrammer** (`agent/agents/telegrammer/telegrammer.py` + `config.yaml`; tool `chat_agent_telegrammer`) — your ONE Telegram send/receive agent, official **Bot API** only (stdlib `urllib`; no third party). Three run-modes: send-once / receive-until-timeout / receive-a-message. KNOT: a bot DMs a user only by **numeric `chat_id`**, or by `@username` **only after that user has messaged the bot first** (else `Bad Request: chat not found`); an optional official **user session** (`api_id`/`api_hash`/`session_string`) reaches private `@username`s a bot cannot cold-DM. Token: `telegram_bot_token` (config.json) / `TELEGRAM_BOT_TOKEN` (data.keys) — Telegram bot tokens do NOT expire.
- **Whatsapper** (`agent/agents/whatsapper/whatsapper.py` + `config.yaml`; tool `chat_agent_whatsapper`) — your ONE WhatsApp send/receive agent, official **Meta WhatsApp Cloud API** only (NO Twilio / TextMeBot / WhatsApp-Web). THE BIG KNOT: a **free-form text message DELIVERS only inside Meta's 24-hour customer-service window** (the recipient must have messaged your number within the last 24 h); a **cold** send needs an **approved TEMPLATE** (`template=hello_world`, or your own template + `template_params`). Meta returns HTTP 200 + a `wamid` (ACCEPTED) **even when it will NOT deliver**, so the agent reports `status: accepted` for free-form (NOT `sent`) and `sent` only for a template. Meta **temporary access tokens EXPIRE (~24 h -> HTTP 401, code 190)** — the durable fix is a permanent **System User** token. Creds: `whatsapp_access_token` / `whatsapp_phone_number_id` (config.json) / `WHATSAPP_ACCESS_TOKEN` / `WHATSAPP_PHONE_NUMBER_ID` (data.keys); RECEIVE needs a public webhook (`verify_token` / `webhook_*`).
- **Instant Messaging Doctor** (`agent/agents/instant_messaging_doctor/instant_messaging_doctor.py` + `config.yaml`; tool `chat_agent_instant_messaging_doctor`) — diagnoses and (with `retry_send: true`) **safely repairs** the two above: validates bot/Meta tokens, contacts, recipient reachability, WhatsApp templates + the 24-h window, and webhook config. `mode` ∈ auto/diagnose/repair/test, `platform` ∈ both/telegram/whatsapp; **non-mutating by default** (`retry_send: false` => it only reports, sends nothing). It is the AUTOMATIC failure companion (a Telegrammer/Whatsapper failure launches it with the same contact/message), the pre-flight you call FIRST for a mission-critical send, and emits `INI_SECTION_INSTANT_MESSAGING_DOCTOR` for Parametrizer/Forker. (The separate long-running **TeleTlamatini** above is the Telegram chat-bridge, not a sender.)

## 11. External MCPs — connect arbitrary MCP servers (config-driven, like Claude Code)
This is your generic MCP-**client** layer: it lets you use the tools of arbitrary external MCP servers (Roblox Studio, GitHub, Steam, Blender-MCP, …) declared in a JSON file — the same `mcpServers` shape a Claude-Code `.mcp.json` uses — with **no hardcoding**. It is distinct from your two built-in context MCPs (System-Metrics / Files-Search, §3) and from ACPX (which spawns coding-agent **CLIs**, not MCP servers). User-facing surface: the navbar **External ▸ MCPs** menu.

**The "hundreds of servers" contract (why it scales):**
- A **catalog** of (potentially hundreds of) servers lives in `external_mcps.json`.
- Only the **ACTIVE set — capped at 5** — is ever spawned/connected, so you never flood the model with hundreds of tools and never spawn hundreds of subprocesses. The rest sit inert in the catalog as text until activated.
- Active servers connect **lazily and once**; their tools are listed + cached and tool calls reuse the live child. A server that fails to connect is skipped for a **60 s cooldown** (negative cache) so a down/unreachable active server can't stall every chat build with connect timeouts.

**How it works end-to-end (the lifecycle):**
1. On each chat build, `tools.py::get_mcp_tools()` calls `external_mcp_manager.get_external_mcp_tools()` right before it returns. Fast path: if nothing is active it returns `[]` instantly — zero overhead, behaviour identical to before the feature.
2. For each active server: connect it over its **transport** — `stdio` (spawn a local child), **`streamable-http`**, legacy **`sse`**, or **`websocket`** (reach an already-running server over the network) — do the MCP `initialize` handshake → `tools/list`, and wrap **each remote tool as a LangChain tool** named `ext__<server>__<tool>` whose body issues a JSON-RPC `tools/call` to the live connection. The transport is auto-detected from the spec (`command` ⇒ stdio; `ws://`/`wss://` ⇒ websocket; `http(s)://` ⇒ streamable-http, or sse when the url/flag says so).
3. Those wrapped tools join your bound Multi-Turn tool surface, so in an operator run you call them like any other tool, alongside your built-in agents. Each is state-changing, so **Ask-Execs gates every external tool call** like any other.
4. **Supervisor + diagnostics:** eight always-on tools manage the layer regardless of which `ext__` tools are bound — `external_mcp_status` (health/tool-count/PIDs/stderr), `external_mcp_reconnect`, `external_mcp_doctor` (static setup triage — transport, runtime, command-on-PATH, placeholder secrets, blockers, next step), `external_mcp_list_tools`, `external_mcp_call` (generic dispatcher for a tool that wasn't directly bound), `external_mcp_import` (add servers from JSON), `external_mcp_set_active` (activate up to five), and `external_mcp_wait` (block until ready or timeout). The canvas/.flw counterpart is the **MCP Doctor** workflow agent (`agents/mcp_doctor/`, wrapped as `chat_agent_mcp_doctor`). A server that connects but exposes 0 tools is auto-relisted every turn and reconnected after a grace window.

**Its elements (where each part lives — all under `agent/` unless noted):**
- `external_mcp_manager.py` — the whole engine: catalog read/write (`load_catalog` / `save_catalog` / `list_catalog`), `set_active` (enforces the 5-cap), `import_servers` (drag-drop merge), the transport clients (`_StdioMcpClient` for a local child + the network family `_StreamableHttpMcpClient` / `_SseMcpClient` / `_WebSocketMcpClient`, all sharing the `_NetworkMcpClientBase` MCP handshake and the SAME public surface so the supervisor treats every carrier identically; `_make_client` / `_connect` dispatch by transport), tool wrapping (`_args_model_from_schema` builds a pydantic args schema from each remote tool's `inputSchema`; `_safe_tool_name`; `_build_tool`), the 5 supervisor tools (`_build_supervisor_tools`), the static doctor (`diagnose_server`), and `get_external_mcp_tools()`. stdio is stdlib-only; the network transports use `httpx` (HTTP/SSE) and `websockets` (both already in requirements). Fully defensive: connects run on background threads (never on the chat-build path) and it never raises into tool-binding — a bad/unreachable/unsupported server degrades to a catalogued-with-reason entry, never a crash or hang.
- `external_mcps.json` — the catalog, resolved **next to `config.json`** (install root in frozen mode, `agent/` in source). Shape: `{"mcpServers": {<key>: {command, args, env, cwd | url}}, "active": [<key>, …]}`. It is **user state** → preserved across self-update and redacted in the self-modify snapshot (see below). Missing file ⇒ fail-open to an empty catalog.
- Endpoints (`views.py` + `urls.py`): `GET /agent/external_mcps/` (catalog for the dialog), `POST /agent/external_mcps/activate/` (persist the active ≤5), `POST /agent/external_mcps/import/` (merge a dropped JSON).
- Frontend: the **External ▸ MCPs** navbar entry in `templates/agent/agent_page.html` → `OpenExternalMcpsDialog` in `static/agent/js/external_mcps_dialog.js`; styling in `static/agent/css/external_mcps_dialog.css` (mirrors the dark tools/skills dialog identity).

**How the user uses it:**
- **External ▸ MCPs** opens the "Activate MCPs" dialog: a searchable list of the catalog, tick up to **5** to activate (a 6th is blocked with a warning), a live summary table of the active ones, then **Continue** saves the active set. Only active servers connect and reach you (the model); the rest stay catalog-only.
- **Drag-and-drop import:** dropping an MCP `.json` (mcpServers shape) anywhere on the page merges it into the catalog after a **confirm** (the file carries an executable `command`, so it is never imported silently). It is *added*, not auto-activated — the user still ticks ≤5.
- Then, in a Multi-Turn run, the user can ask you to use that server's tools together with your built-in agents — e.g. *"using the Roblox MCP, scaffold a Mario template in the open place, then write the report with File-Creator"*.

**How to modify the code about it:**
- Active cap → `MAX_ACTIVE` in `external_mcp_manager.py`.
- Supported carriers → `_SUPPORTED_TRANSPORTS` = `{stdio, streamable-http, sse, websocket}`. To add one more (e.g. `tcp` / `named-pipe`, currently catalogued-only), write a client subclassing `_NetworkMcpClientBase` (implement `_open` / `_rpc` / `_notify` / `close`) and wire it into `_make_client`. Loopback round-trip tests live in `test_external_mcp_transports.py`.
- Tool naming / schema mapping → `_safe_tool_name` / `_args_model_from_schema` / `_build_tool`.
- Dialog UX → `external_mcps_dialog.js` (+ `.css`); the menu entry in `agent_page.html`.
- **Carriage is already wired — keep it coherent:** `build.py` ships `external_mcps.json` to the install root; `apply_update.ps1 $Preserve` + the `self_update.py` docstring preserve it across updates (so a user's added servers + active picks survive); `copy_source_assets.py` redacts it (a server `env` may hold API keys) and guarantees `external_mcp_manager.py` + `external_mcps.json` via `REQUIRED_SNAPSHOT_FILES`. After any change here, re-run the two inclusion sweeps (§9 quality gates: `tlamatini-self-update-inclusion` / `tlamatini-self-modify-inclusion`) until clean.
- **Trust note:** an activated external server hands arbitrary tools to you — the same trust surface as a Claude-Code MCP. It is user-chosen and Ask-Execs still gates each call.
